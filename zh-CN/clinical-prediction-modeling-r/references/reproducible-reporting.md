# 可复现执行、批量模型与研究报告

目标是让另一位研究者在干净会话、同一受控输入和记录的包版本下重新生成分析数据、模型、表、图和报告。可复现不等同于无偏倚；它使偏倚处理、版本和决策可以审计。

## 1. 从干净会话运行

使用脚本而不是交互工作区作为事实来源。把导入、派生、建模、验证和报告拆成顺序明确的脚本；每一步显式读取上一步的受控输出或调用函数文件。不要用 `.GlobalEnv` 中碰巧存在的对象作为输入。

```sh
Rscript --vanilla scripts/01_import.R
Rscript --vanilla scripts/02_derive_analysis_data.R
Rscript --vanilla scripts/03_develop_validate.R
Rscript --vanilla -e 'rmarkdown::render("reports/model-report.Rmd")'
```

脚本从项目根目录运行，并只使用相对路径，例如 `file.path("data", "raw", "baseline.csv")`。在脚本开始处设定所有随机种子；不在分析/报告执行中调用 `install.packages()`。当使用并行或随机调参时，记录随机数策略、工作线程和种子。

## 2. 固定 R 包环境与输入版本

对需要复现或共享的项目使用 `renv`。首次设置后提交 `renv.lock`（不提交受试者级原始数据），在另一台电脑用 `renv::restore()` 恢复；代码可用后才以 `renv::snapshot()` 更新锁文件。交付前运行 `renv::status()`，并解释任何未同步状态。

```r
renv::init()       # 每个项目仅在建立阶段执行一次
# 安装/更新所需包并验证全部脚本后：
renv::snapshot()
renv::status()
```

包锁定仍不足以完全重现计算：报告 R 版本、操作系统、外部系统依赖和渲染工具版本。记录实际输入文件的名称、日期、不可识别化版本 ID 或校验和；不要在公开报告中写入可识别路径或原始患者 ID。

```r
input_files <- c("data/raw/baseline.csv", "data/raw/outcomes.csv")
run_manifest <- data.frame(
  file = input_files,
  md5 = unname(tools::md5sum(input_files)),
  stringsAsFactors = FALSE
)
write.csv(run_manifest, "outputs/run_manifest.csv", row.names = FALSE)
capture.output(sessionInfo(), file = "outputs/sessionInfo.txt")
```

校验和只能证明文件字节是否相同；还需保留数据字典、纳排标准、查询/抽取日期和去标识策略。

## 3. 函数化与批量模型

重复的导入检查、时间窗口、模型拟合和性能计算写成输入/输出明确的函数。函数名称说明动作，参数说明数据；在函数内检查列、长度、编码和返回值。用小型合成数据测试边界情形（缺失键、重复键、零事件、非预期水平），而不是只测试“正常”数据。

对于候选模型、中心或插补数据集的批量运行，保存模型注册表，而不只保存最后一个对象：每一行应包括模型 ID、公式/超参数、训练数据版本、重抽样方案、随机种子、成功/失败状态、警告摘要和性能结果。`purrr::safely()` 可保留个别失败，但不能静默丢弃失败模型；报告失败数量、原因和受影响的输入。

`broom::tidy()`、`broom::glance()` 和 `broom::augment()` 适合把系数、整体摘要和行级诊断整理成可审计表。对 `rms`、`survival`、`riskRegression` 等对象，先核对该包的当前 `broom` 支持；若不完整，显式提取并标记对象/函数版本，不能假定所有模型都有相同字段。

## 4. 预处理与验证的执行边界

将数据质量规则和研究设计固定在数据派生阶段，但把任何需要从数据估计参数的操作放在训练与重抽样中：插补、中心/尺度、样条结点（若由数据确定）、降维、变量选择、特征工程、类别平衡和调参。使用 `recipes`/`workflows` 时，让工作流在每个训练折拟合 recipe，再将已训练规则应用到验证折；或在手写代码中实现等价边界。

固定时间/地域外部验证集时，开发脚本不读取其结局来作选择。只有在模型及其预测时点、参数和阈值锁定后，才计算外部验证性能。

## 5. R Markdown、Quarto 与最终报告

R Markdown (`.Rmd`) 和 Quarto (`.qmd`) 都可将叙事、代码、数值、表格和图形绑定为同一源文件。报告应从干净会话完整渲染，正文数值使用内联代码或由同一对象生成，避免手工复制后与表格不一致。

在质量控制阶段不要全局隐藏错误、警告或消息。错误应使最终分析构建失败；只有确认预期且无临床/统计含义的消息才可在正式受众版本中有针对性地隐藏。图表代码显式设定尺寸、标签、单位、图例、调色和输出路径；同一指标在图、表和正文中使用同一时间点、样本和置信区间定义。

最小报告结构：

1. 问题、目标人群、预测时点、预期用途与临床行动；
2. 数据来源、数据契约、纳排标准、结局/预测因子、缺失与时间审计；
3. 模型开发、所有预处理、样本量和验证方案；
4. 最终模型、性能/校准/效用、内部或外部验证与不确定性；
5. 限制、适用范围、不可用场景、TRIPOD+AI 与 PROBAST 自查；
6. 运行清单：代码版本、输入版本、包锁文件、随机种子、R/系统/渲染信息。

输出 Word/PDF/HTML 前逐项检查：无隐藏的错误对象、无占位图、无开发集指标被误标为外部验证、无患者级数据或本地绝对路径泄露、无与最终对象不一致的手工文字。

## 6. 分析式笔记本纪律

探索性笔记本也是研究记录，不是一次性草稿：

- 用描述性标题、ISO 8601 日期（`YYYY-MM-DD`）和第一段目的说明，便于多年后检索；
- 记录为什么选择或放弃一种处理/模型，包括失败的尝试，而不是只保留最终成功路径；
- 发现原始文件错误时，用脚本生成修正并说明依据；不要直接覆盖原始文件或在 Excel 中手工改值；
- 结束一个工作阶段前从干净会话完整渲染，必要时清除缓存；缓存只能加速，不可替代完整重跑；
- 为长期复用的 notebook 记录包版本、R/渲染环境和输入数据版本，并统一命名、归档在项目目录中。
